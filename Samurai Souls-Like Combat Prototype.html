Samurai Souls-Like Combat Prototype (WIP)

Description

  • Kaleb Taylor

  • Role: C++ Gameplay, Animation Blueprints, Enemy AI

  • Status: Work In Progress

Project

This prototype explores combat architecture, animation timing, and AI decision-making in Unreal Engine 5. The system is driven by C++ and focuses on coordinated character state, stamina-gated actions, lock-on targeting, and enemy behaviour that reacts to player spacing and pressure rather than running through a fixed sequence.

My Work

My Role - Gameplay Programmer (Solo)
  • Designed and implemented the full combat stack in C++: state-driven locomotion, light/heavy attacks, dodges, blocks/parries, stamina-gated actions, and weapon archetypes with distinct move sets and timing profiles.
  • Built a combat state machine that owns action eligibility, input buffering, and clean transitions between locomotion, attacking, recovery, and stun, keeping edge cases out of Animation Blueprints and centralizing gameplay rules.
  • Authored enemy AI using Behavior Trees and Blackboards that react to player distance, recent actions, and the AI's own stamina, driving distinct branches for pressure, spacing, and punish logic instead of a fixed attack loop.
  • Integrated Animation Montages and notifies to control hit windows, root motion, and reaction timing, with gameplay logic in C++ and timing data in animation assets for designer-friendly tuning.
  • Implemented a progression layer where players earn a reusable currency from encounters and spend it to level core stats, scaling survivability and damage output in a Souls-like loop while preserving risk on death.
  • Built an inventory and equipment system that tracks items, weapons, and runes, applies stat modifiers and requirements, and exposes data-driven parameters so new gear and weapon types can be added without changing core code.
  • Implemented lock-on targeting, character rotation, and camera handling so strafing, spacing, and hit reactions remain readable and weighty during duels, supporting both free-movement exploration and tight 1v1 encounters.

Technical Breakdown

A. C++ Combat State Machine

C++-Driven Combat States
  • Implemented a lightweight combat state machine in C++ to handle idle, attacking, dodging, stunned, and dead states.
  • Used clear enums and helper functions instead of burying logic in Blueprints, so behaviour is easy to trace and debug.
  • Added input buffering so attack inputs during animations are stored and played at the next valid window, making combos feel responsive instead of spammy.

B. Stamina & Risk/Reward

Stamina & Resource System
  • Built a stamina system in C++ that gates attacks, dodges, and blocks.
  • Tuned regen rules (e.g., faster out of combat, slower when attacking) to create a simple risk/reward loop.
  • Exposed key values to data tables / Blueprint variables so designers (or future me) can tune balance without touching C++.

C. Lock-On & Camera Behaviour

Lock-On Targeting + Camera
  • Implemented a soft lock-on system that picks the best target based on angle and distance.
  • Rotates character movement relative to the locked target so strafing and circling feel natural.
  • Added small touches like camera offset, slight FOV changes during attacks, and camera clamping to keep the action readable.

D. Animation Integration

Animation-Driven Combat
  • Used Animation Montages with notifies to open/close hit windows, spawn traces, and trigger sound / VFX.
  • Kept gameplay logic in C++ but drove timing from animation data, so designers can tweak attacks by editing animations instead of code.
  • Set up a clean separation between movement animations (locomotion state machine) and combat montages so the character never "breaks" visually.

E. Enemy AI & Telegraphed Attacks

Enemy AI with Behaviour Trees
  • Created a simple enemy archetype that patrols, detects the player, and picks different attack patterns based on distance.
  • Used Blackboard keys to store target location, distance bands (close/mid/far), and current attack choice.
  • Designed animations and delays to telegraph attacks clearly, making it feel fair and learnable, like a Souls-style fight.

F. Game Feel & Polish

Game Feel Systems
  • Added hit stop, camera shake, and brief slow-down on big hits to sell impact.
  • Implemented directional hit reactions (front, back, left, right) so the character reacts based on where damage came from.
  • Tweaked movement acceleration and rotation rates per state (locked-on vs free movement) to keep the character feeling responsive but grounded.
